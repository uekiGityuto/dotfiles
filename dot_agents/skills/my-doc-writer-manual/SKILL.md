# ドキュメント作成

ドキュメントは、読者が目的に沿って判断・作業・確認できる形で残す。説明量ではなく、目的に必要な情報が迷わず読めることを優先する。

## 基本方針

- 目的を先に決める。何を判断してほしいのか、何を実行してほしいのか、何を共有したいのかを明確にする。
- 読者を決める。読者が使わない背景、技術詳細、内部事情を本文に増やさない。
- 簡潔にわかりやすく書く。長い説明は、目的に必要な情報だけに絞る。
- 1 つの文に複数の指示、判断、理由を詰め込まない。混ざる場合は文を分ける。
- 目的に関係ない補足を足さない。大事な詳細でも目的が違うなら別資料に分ける。
- 事実、推測、判断、残作業を混ぜない。
- 表記揺れを潰す。人、組織、システム名、画面名、機能名、データ名は同じ呼び方に揃える。
- 目線を揃える。実行する側、実行される側、保存する側、参照する側、確認する側を同じ表や説明の中で混ぜない。
- 読者が判断に使わない自明な説明や、他の箇所や資料の目的から明確に分かることは書かない。
- 伝えたいことは一箇所にまとめる。似た説明を複数箇所に散らさず、正本を決めて必要な場所から参照する。
- 複雑な内容や複数の内容を一つの塊で説明しない。必要に応じてファイル、セクション、表、図に分ける。図が有効なら Mermaid 記法などを使う。
- 未決事項や残作業がある場合だけ、本文に散らさず一箇所にまとめる。
- リンクや参照には、個人のローカルパス、ユーザーディレクトリ、gitignore されているディレクトリを正本として使わない。必要な場合は、共有できる参照方法をユーザーに確認する。
