# 調査資料作成

調査資料は「何が分かっているか」「何が決まっていないか」「次に何をするか」を、読者が迷わず判断できる形で残す。装飾的な説明を避け、根拠・判断・残作業を分ける。

## 基本方針

- 簡潔に書く。説明が長くなる場合は、何を判断してほしいのかを先に決め、目的に必要な情報だけに絞る。
- 資料の目的に合わない説明を入れない。大事な詳細でも目的が違うなら別資料に分ける。
- 資料の目的に関係ない補足を足さない。たとえば取得時刻、件数、運用背景などでも、今回の論点に効かないなら削る。
- 事実、推測、判断、残作業を混ぜない。
- 表記揺れを潰す。人・組織・システム名・テーブル名・カラム名・画面名は同じ呼び方に揃える。
- 用語は一時的な用途ではなく、資料全体での扱いに合わせる。検証中だけの呼び名、移行中だけの呼び名を、恒久的な対象名として使わない。
- 目線を揃える。実行する側、実行される側、保存する側、参照する側、確認する側を同じ表や説明の中で混ぜない。
- 読者が詳しくない技術のコードや内部 API をそのまま出さない。条件、入力値、出力値、保存先、画面上の動き、運用上の影響に置き換える。
- 1 つの文に複数の指示、判断、理由を詰め込まない。混ざる場合は文を分ける。
- 伝えたいことは一箇所にまとめる。似た説明を複数箇所に散らさず、正本を決めて必要な場所から参照する。
- 読者が判断に使わない自明な説明は書かない。
- 資料の目的に関係する不明点や曖昧な点は、調査できるなら調査し切る。理由があって調査できないものだけを、資料末尾の残作業に集約する。
- 本文中に `要調査`、`未確認`、`不明` と書いて終わらせない。本文には分かっている事実と根拠を書き、調査できないものは残作業に移す。
- 複雑な内容や複数の内容を一つの塊で説明しない。必要に応じてファイル、セクション、表、図に分ける。
- データの流れ、参照関係、処理順、状態遷移は、文章や表で読みにくい場合に図にする。必要なら Mermaid 記法を使う。

## 作成手順

1. 目的を一文で決める。
   例: 「実装方針を決めるための現行仕様調査」「外部確認が必要な残作業の一覧」「データ移行時の変換ルール整理」。

2. 読者を決める。
   例: 実装者、レビュー担当、運用担当、顧客担当、外部ベンダー。読者が読まないコード表現は避ける。

3. 根拠を確認する。
   コード、DB、ログ、設定、管理画面、現行画面、仕様書、チケット、ヒアリング結果のどれを見たかを分ける。資料の目的に必要な範囲で、確認できるものは確認し切る。確認できないものは残作業に入れる。

4. 切り口を決める。
   表の行と列は「伝えたい判断」が伝わる単位にする。確認先、業務領域、機能、データ、画面、環境などの切り口を混ぜない。

5. 重要な情報を左に置く。
   読者が最初に見るべき条件、対象、判断を左側に置く。補足、備考、根拠は右側に置く。

6. 最後に未決事項を集約する。
   個別ファイルには根拠を書く。本文中に要調査を散らさず、調査できなかったものだけを、未決事項、ヒアリング事項、残作業として正本となる一箇所へ集める。

## 前提・用語定義

似た名前だが意味が違うデータソース、環境、画面、時点、資料種別は、資料の冒頭で定義する。

- 共通前提は親 index や冒頭の共通セクションに置き、個別ファイルで同じ説明を繰り返さない。
- 名称を統一するだけで済むものは用語定義を増やさない。読み手が混同しそうな概念だけ定義する。
- 「A から抽出した B」のような表現は、A と B が同じデータなのか、一部だけ取り出した別データなのかが分かるように書く。
- チーム共有用資料では、個人環境のパスや個人フォルダの URL を正本参照にしない。共有ストレージ、チケット、仕様書 URL など、他の読者が追える参照へ置き換える。gitignore された個人メモやローカル調査メモはこの限りではない。
- 参照したいファイルが個人フォルダにしか見えていない場合は、共有資料にそのまま書かない。共有できる取得元や相対パスの書き方をユーザーに確認する。

## 方針変更・追加回答の反映

調査途中の仮説や選択肢は、後から回答や方針が確定したら整理し直す。

- 方針が確定したら、古い選択肢表、交渉案、暫定案を残しっぱなしにしない。履歴として必要なら現在の結論に必要な範囲だけ残す。
- 「A も可能」「B も可能」だけで終わらせず、今回採る方針を明示する。方針と代替案を同列に置かない。
- 短期的な回避策と、本番で採る方式を分ける。回避策を書く場合は「何を確認できるか」「何は確認できないか」まで書く。
- 日付や期間は年を入れる。`10月中旬`、`2月頃` だけにせず、`2026年10月中旬`、`2027年2月頃` のように書く。
- 期間がスケジュールに依存する場合は、根拠となる計画やチケットへの参照を添える。

## 表の設計

表は切り方が最重要である。1 つのセルに複数の意味が混ざったら列を分ける。

列の例:

- `条件`
- `読む値`
- `書き込む値`
- `参照元`
- `識別子 / キー`
- `現行で分かっていること`
- `対応方針`
- `進め方`
- `完了条件`

表を作る時は次を確認する。

- 行と列の粒度が揃っているか。テーブル単位、カラム単位、画面単位、確認先、用途、判断を混ぜない。
- どのテーブル・ファイル・画面の話かが、表の直前または列名で分かるか。
- 重要な判断が左側にあるか。
- セルが長文になっていないか。長ければ列分割、表分割、別節化を検討する。
- 判断を書く場合、判断理由も同じ行か近くに置く。数字だけ、状態だけ、印象だけで判断を終わらせない。
- `備考` に判断、条件、SQL、例外を全部入れない。

## 件数・サンプル・根拠の書き方

件数は、何を数えた数字かが分かるように、必要に応じて集計元と範囲を添える。

- 調査対象全体の件数なのか、一部データの件数なのかを明記する。
- 推定値、概算、実測値、代表例を区別する。
- 代表例や一部確認を、全量調査のように見せない。
- 数字だけで判断を書かない。なぜ対応方針や未決事項につながるのかを書く。
- 同じ件数が複数箇所で出る場合、正本を決める。

## 分ける単位

分野の違うものは分ける。大きな表が読みづらい場合は、分類軸を作り直す。

- 機能ごとの差
- 画面ごとの差
- データ種別ごとの差
- 外部連携ごとの差
- 設定・権限・運用条件
- テーブル / カラム / キー / 設定値のカタログ
- 残作業・未決事項

「その他」は最後の手段にする。その他が膨らんだら分類軸を作り直す。

## 技術表現・コード

読者が詳しくない技術は、その技術名や関数名だけで説明しない。

- 関数名、クラス名、フレームワーク API だけで説明しない。画面上の動き、DB に入る値、参照条件、入出力に置き換える。
- フレームワークやプロダクトの内部用語は、必要なら短く説明する。
  例: `メタ情報` より `対象データに追加で保存される属性`。
- 一般名詞に見える DB カラムや設定名は、何が入るかを補足する。
  例: `content` だけではなく `本文。HTML や埋め込み記法を含む場合がある`。
- 読み手が誤解しやすい内部用語や状態値は、初出または表の近くで短く補足する。
- 似た識別子や関連値が複数ある場合は、「何と何が紐づくか」を文章だけで済ませず、必要に応じて表で整理する。
- SQL やコードを書く場合は文章と分け、読めるように改行とインデントを揃える。

## 用語と表現

曖昧な表現と難しい表現は別の観点として確認する。

曖昧な表現を避ける。

- `本番相当データ` のような表現は避ける。対象の環境、データ範囲、取得時点を明確にする。
- `一覧表示`、`詳細表示` だけではなく、どの画面で何を表示する場合かを具体化する。
- `添付ファイル`、`外部データ`、`設定値` のような広い語は、作成元、保存値、参照方法を補足する。
- `別枠で扱う`、`個別扱い`、`良さそう`、`未確定` のような表現だけで終わらせない。何と分けるのか、何が決まれば判断できるのかを書く。
- 可能性を書く場合は、成立条件を書く。条件が分からないなら未決事項にする。
- `最新`、`本番由来`、`サンプル` などは時点と範囲を添える。
- `検証用`、`暫定`、`現行`、`新` のような語は、用途・期間・切替後の扱いが変わると誤解を生む。最終的に使い続けるものは `検証用` と呼ばない。
- `可能` と書く場合は、契約上可能なのか、技術的に可能なのか、運用上許容されるのかを分ける。

不要に難しい表現を避ける。

- 必要な用語は使う。ただし、意味を伝えるうえで重要ではない専門語・硬い表現をあえて使わない。
- `非空値` より `データがある / データがなかった`。

## リンク・参照

URL やファイル参照は、読者がすぐ確認できる形にする。

- 裸 URL を本文に置かず、Markdown リンクにする。リンクラベルはページ名、資料名、画面名、管理 ID など、開く前に意味が分かる名前にする。
- ローカルファイルや別資料へのリンクは、相対パスが実際に解決できるか確認する。
- 共有資料に入れる参照と、ローカルメモに留める参照を分ける。内部情報は、共有範囲と必要性を確認してから載せる。
- 個人フォルダに展開した共有ソースを参照する場合は、個人ローカルパスではなく、共有できる取得元と取得後の相対パスを書く。書き方が分からなければユーザーに確認する。
  例: `existing-implementation/nsj2022-main/src/wp-content/chart/news.php`。ソースコードは Backlog の該当 issue から取得できる。
- 参照リンクが複数ある場合は、単なるリンク集にせず「読むべき資料」「補足資料」「代替サービス」のように扱いを分ける。
- 回答やチケットで得た情報は、資料本文にも参照元として残す。後から読んだ人が「誰の回答に基づく結論か」を追えるようにする。

## 検証・実装向けメモ

調査資料が後続の実装や構築のインプットになる場合は、実装者が判断や作業に使う情報だけを残す。

- 構築時に必要な作業、開発中の動作確認、検証環境での確認、本番切替時の扱いを分ける。
- 開発中に外部接続できない、または避けるべき場合は、代替データやテスト用の応答で何を確認するかを書く。
- 現行環境を使った補助確認を書く場合は、既存運用や本番処理に影響しない条件を添える。
- 実装者が使わない背景を入れすぎない。実装・構築で必要な作業、決めること、注意点に絞る。

## 未決事項・残作業

未決事項は一箇所に集約し、作業として着手できる形にする。資料の目的に関係し、調査できるものは資料作成中に調査し切る。アクセス権限、環境、外部回答待ち、時間制約などの理由で調査できないものだけを残作業にする。

基本列:

| 残作業 | 進め方 | 完了条件 |
|---|---|---|

必要に応じて `優先度` や `領域` を足す。残作業が少ない場合や分類しなくても読める場合は増やさない。

進め方は主な確認先または決定工程を短く示す。

- 顧客または業務担当に確認
- 現行データまたはログで確認
- 保守担当者または運用担当者に確認
- 実装方針または移行計画を策定
- 詳細設計で確定

残作業の内容から確認事項が分かる場合は、進め方に同じ説明を繰り返さない。進め方と完了条件で十分に伝わることは、別列や補足説明で繰り返さない。

## レビュー観点

提出前に次を確認する。

- 同じ事実を複数ファイルで違う言い方にしていないか。
- 既に別ファイルに詳しく書いた内容を、本文で長く再説明していないか。必要ならリンクにする。
- 似た説明が複数箇所に散っていないか。正本にまとめられないか。
- 判断に使わない件数やサンプルが増えていないか。
- 資料の目的に関係ない時刻、件数、背景、細部が残っていないか。
- 共有資料に個人フォルダのパスや、他の読者が辿れないリンクが残っていないか。
- 「未確定」「要調査」「未確認」が本文中に残っていないか。調査できるものは調べ切り、調査できないものは残作業に集約しているか。
- 技術を知らない読者が読んでも、何のデータが何に使われるか分かるか。
